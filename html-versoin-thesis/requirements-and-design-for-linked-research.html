Introduction

For over two computer decades we have imitated paper on screens (extremes being Adobe Acrobat and the HTML page) [1]. This citation from Ted Nelson’s paper “Xanalogical Structure, Needed Now More than Ever: Parallel Documents, Deep Links to Content, Deep Versioning and Deep Re-Use” from 2000-05-23 is still true for today.

PDF documents are familiar and easy to read for humans. But they are also antithetical to the spirit of the Web, lacking user interactivity and being difficult for machines to read [2] according to “Adventures in Semantic Publishing: Exemplar Semantic Enhancements of a Research Article” by David Shotton et. all.

Tim Berners-Lee, working at CERN in 1989, needed an information system that would help scientists to create and discover each others documents across systems. In May of this year, he made a proposal for an information management system; todays World Wide Web [3].

The Web today contains interlinked hypertext documents as proposed in 1990 by him and Robert Cailliau. Their idea of having one-directional hyperlinks between documents and accessing information as a web of nodes [4] has emerged as a large universe of documents, where ideas can be exchanged, cat videos watched and knowledge shared between people or institutions.

However, most data and knowledge on the Web is represented in a unstructured machine-unfriendly way and on top of that, is not provided to humans with the fullest possibilities the World Wide Web offers. This is also true for publications in the Web Science community as it continues to conduct research tracks, workshops, and challenges, in a manner that even the call for papers of top-level Web conferences request research documents that are neither machine nor Web-friendly [5]. Submissions are often solely demanded in Word or PDF formats, where it is challenging to extract information from within [5].

The reason for having PDF or Word documents for publications in the Web Science area lies in the nature of the publication process. According to “Call for Linked Research”, this process is described as follows: researchers need to have read and reuse access to other researchers work, and adopt archaic desktop-native publishing workflows. Publishers try to remain as the middleman for society’s knowledge acquisition [6].

Problem Statement

Through movements like Open Access, data and research publications have become more accessible for the general public. But Sarven Capadisli et. all state in their publication Enabling Accessible Knowledge the following: While there are notable efforts making research publications and data available for free with open access, there is a self-evident disconnect between research documents as far as machines are concerned [5]. According to them, research documents are not as human and machine friendly as they could be, making it unpleasant to us – but also for the machines – to explore research documents on the Web.

As reported by Frank van Harmelen and Dieter Fensel in their publication Practical Knowledge representation for the Web, the lack of semantic markup is a major barrier to develop more intelligent document processing on the web. [7]

In order to close the disconnect between research documents, Sarven Capadisli is in progress to develop a tool to share and reuse research knowledge by employing the native Web technology stack. The project called Linked Research will enable researchers to take full control, ownership and responsibility of their own knowledge, making it possible to get away from the imitation of paper and use the Webs current possibilities to explore hypertext documents in a user-friendly way [5].

Goal

As part of a possible solution to make research knowledge more accessible and more pleasant for people and machines, Linked Research is going to implement an authoring tool to create and edit research papers directly within a browser. In addition it will enable the user to author triple statements to enrich the document with markup.

The objective of this thesis is to establish requirements for Linked Research, compare editing libraries and provide an overview of RDFa Tools available on the market.

Background

This chapter is intended to provide background information and define terms to understand this thesis.

HTML

The Hypertext Markup Language (HTML) is the core language of the World Wide Web (WWW) and currently released as version 5.0. The language is used to create web pages and is recommended by the main international standards organization for the WWW (W3C, World Wide Web Consortium) [8]. HTML specifies the structure of a website with tags (like html ). The following listing shows the basic structure of an HTML 5 document: 

<!DOCTYPE html>
<html>
<head>
<title>Title of the Web page</title>
</head>
<body></body>
</html>
[9]

The DOM (Document Object Model)

The DOM is a language neutral interface, which allows scripts to dynamically access and change the structure of an (HTML)-document. With the DOM it is possible for scripting languages like JavaScript to access HTML elements and rearrange, delete or add new elements to the HTML-document. The DOM makes DHTML (Dynamic HTML) possible [11].

Web Browsers

Web browsers (or just browsers) are used to represent web pages on the World Wide Web. They provide the user interface for web applications. The current major browsers on the market are Firefox, Internet Explorer, Google Chrome, Opera and Safari with their corresponding web browser engines Blink (Google Chrome and Opera) Trident (Internet Explorer), Gecko (Firefox) and Webkit (Safari). The web browser engine is a software component that is responsible to display content on the screen [12].

What you see is What you…

WYSIWYG

WYSIWYG stands for “What-You-See-Is-What-You-Get” and describes an editing system, in which a user’s edited content is displayed in a form close to the final representation of the text, while he is writing the text [10].

WYSIWYM

WYSIWYM is an acronym for “What-You-See-Is-What-You-Mean”. a WYSIWYM editor tries to capture the separation of presentation and content when writing a document. The focus of a WYSIWYM-editor lies on the preservation of the meaning of an element within a text instead the presentation of the element in the document [10].

Linked Data

Linked Data is part of the Semantic Web and describes free available structured data reachable through Uniform Resource Identifiers (URI). It is published in a format (often RDF) so that it can be interlinked with other data, and through that, be more useful. [13].

Time Bernes-Lee published on 2006-07-27 an article called „Linked Data – Design Issues“. In this publication Bernes-Lee addresses four rules for Linked Data:

  1. Use URI as names for things.
  2. Use HTTP URIs so that people can look up those names.
  3. When someone looks up a URI, provide useful information, using the standards (RDF, SPARQL).
  4. Include links to other URIs. So that they can discover more things.
[13]

HTTP URI

URIs build the basis of Linked Data and are used as names for things [13]. They are a sequence of characters to identify an abstract or physical resource on the Web [14]. The HTTP scheme is the most common used form of URIs and is used since 1990 by the World-Wide Web global information initiative [15]. They provide the user a simple way to explore the Web and its documents (resources).

RDF

RDF is a resource description framework to express information about resources. The framework is specified by the World Wide Web Consortium (W3C) and lets anyone make any statement about anything [16]. A statement is composed of three atomic parts (subject, predicate and object). Those three parts together are called a triple. The subject (line 1 of listing 2) says what the statement is about and the predicate (line 2 of listing 2) describes the relationship to its object (line 3 of listing 2) resource. Typically triples are described by HTTP URIs [17]. The following triple statement gives an example: 

<http://renatostauffer.ch/i#>
<http://xmlns.com/foaf/0.1/knows> 
<http://csarven.cs/#i>

The statement from above says that the thing <http://renatostauffer.ch/i#> knows the thing <http://csarven.cs/#i> . Those resources (things) can now be further described through RDF vocabularies, which provide semantic information about these resources [18]. As the example from above shows, predicates are vocabulary terms that are used to relate the subject with an object. The predicate itself is an HTTP URI, which has a description for the term in use when dereferenced.

RDFa

Resource Description Framework in Attributes or simply RDFa is a W3C recommendation that enables authors of HTML, XHTML and other XML-based document types to enrich their documents with metadata. RDFa provides a set of attributes to describe elements in XML languages [19]:

about
Defines what the resource is about (subject in a triple statement) [20].
rel / rev
Describes the relationships between resources (predicate in a triple statement) [21].
src / href / resource
Those three attributes are used to state the object of a triple statement. While the resource-attribute is intended to state a non-navigable partner resource of a subject [22], the href-attribute is used in conjunction with navigable resources [23]. The src-attribute is used, when a resource is embedded [24].

SPARQL

SPARQL is a recursive acronym and stands for SPARQL Protocol and RDF Query Language [25]. It is a query language for RDF data and can be used to query RDF triple statements across different data sources [26].

Related Work

The idea of having a publishing tool paired with a triple markup feature is nothing new. Different products, with different focuses have emerged. Three of those products are RDFaCe, Loomp and Wordlift, which will be introduced in the following sections.

RDFaCe

RDFaCe is a semantic content authoring (SCA) tool which integrates four different views on the semantically annotated content. Those views are synchronized, which means that when a change in a view happens, the other views apply the change. The focus of RDFaCe lies on usability by implementing the four views for different personas involved in the content creation life-cycle.

The four views RDFaCe implements are [27]:

WYSIWYG View
Used to create text by authors, journalists etc.
WYSIWYM View
Extension for the WYSIWYG View.
RDF Triple View
Useful for curators and authors for verifying that entities and facts were correctly annotated.
Source Code View
Shows the HTML source code of the article and its RDFa annotations. Useful for software and knowledge engineers.

Loomp

Loomp is a SCA with a one click annotation strategy (OCA) according to “One Click Annotation” by Heese et al. [28]. Loomp addresses the issue for missing tools for creating semantic content by non-domain experts. The main focus lies on simplicity by providing an intuitive interface. The editor allows users to annotate words or phrases with references to ontologies and to create relationships between annotated phrases. Loomp relies on a server to handle and process the semantic content [29].

Wordlift

Wordlift is a semantic web plugin for the content management system (CMS) WordPress. The user is able to run a semantic web analysis on an authored text to find entities from the Linked Open Data Cloud, that do match words or phrases within the text. After an entity has been found, the user can bind it to the text. It is also possible to manually create entities, which do not exist in the Linked Open Data Cloud yet [30].

If you view this thesis on paper, feel free to visit the following URL to see Wordlift in action on video: http://renatostauffer.ch/requirements-and-design-for-a-tool-to-author-triple-statements/#wordlift-video

Requirement Engineering

In this section, we will derive functional and non-functional requirements for a Web based authoring tool intended to help researchers to create and consumers to read publications in a user friendly environment. For this propose parts of the requirements engineering framework according to Pohl et al. [31] is used as a reference. The used framework is structured as shown in figure 1 (Fig. 1).

The framework is composed of three parts: The System Context, Core Activities and Requirements Artifact.

Firstly the System Context will be described to identify important characteristics of the system and give the reader of this thesis an overview of the system’s context. Depending on the established context, goals were determined and scenarios to be implemented are worked out. Those requirement artifacts are documented in the Chapters 5.3 Goals as functional and non-functional requirements and 5.4 Scenarios.

System Context

Each software-intensive system lies within a certain context and is through that influenced by the definition of the system requirements [31]. This chapter describes the system context with its facets for Linked Research. The Usage Facet, Subject Facet, IT-System Facet and Development Facet each describe a certain aspect of Linked Research and build the entry point for the systems requirements.

Based on the established Usage Facet a first sketch of Linked Research was drawn:

  1. The user edits or creates text with the editor. When the user selects a text passage or word a popup menu shows up. From there the user has the ability to assign styles to the text (bold text, italic, etc.).
  2. The user can also toggle in / out the meta editor to create triple statements.
  3. The triple store is held locally. The user can insert or delete triple statements.
  4. Triple statements can be fetched to e.g. visualize the current graph.
  5. The meta editor talks to a external service to retrieve suggestions for possible RDF vocabulary terms when creating a triple statement.

Text Editing on the Web

Before considering a particular editor, known approaches to edit text within the browser need to be determined.

A search for the terms “How to write a wysiwyg editor”, “text editor approaches in browsers”, “contentEditable” and “alternatives for contentEditable” on Google.com resulted in three possible approaches on how an author is able to edit text in the browser.

textarea
In the early days of the Web, users were limited to text boxes when it came to online editing. Text boxes (expressed with the <textarea>-element) do not support multiple fonts and styles.
designMode
The designMode attribute of the document object is able to make a whole HTML-page editable by just setting it to true [source http://www.w3.org/TR/html5/editing.html#editing-0].
contentEditable
In contrast to the designMode, the contentEditable attribute makes just some part of the document (like a div-element) editable [http://www.w3.org/TR/html51/editing.html#contenteditable]. Example XY shows how the attribute can be used: 
<div contentEditable="true">
</div>

The <div>-element down below is now editable due to setting the <contentEditable>-element equal to true (pleas visit http://renatostauffer.ch/requirements-and-design-for-a-tool-to-author-triple-statements/#text-editing if you read this thesis on paper).
