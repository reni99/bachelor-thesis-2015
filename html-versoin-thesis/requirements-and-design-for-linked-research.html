Introduction

Nothing is pleasanter to me than exploring in a library. [1] Many years have past by since this quote from Walter Savage Landor. Meanwhile technology has evolved and brought the World Wide Web to life: A massive library full of data, where knowledge is often represented in an unstructured, machine-unfriendly manner. more to come…

Problem Statement

Through movements like Open Access, data and research publications have become more accessible for the general public. But Sarven Capadisli et. all state in their publication Enabling Accessible Knowledge the following: While there are notable efforts making research publications and data available for free with open access, there is a self-evident disconnect between research documents as far as machines are concerned. [2] According to them, research documents are not as human and machine friendly as they could be, making it unpleasant to us – but also for the machines – to explore research documents on the Web.

As reported by Frank van Harmelen and Dieter Fensel in their publication Practical Knowledge representation for the Web, the lack of semantic markup is a major barrier to develop more intelligent document processing on the web. [3]

In order to close the disconnect between research documents, Sarven Capadisli is in progress to develop a tool to share and reuse research knowledge by employing the native Web technology stack. The project called Linked Research will enable researchers to take full control, ownership and responsibility of their own knowledge, making it possible to like-minded people as Walter Savage Landor to explore the giant library called “Web” in a user-friendly way. [4]

Goal

As part of a possible solution to make research knowledge more accessible and pleasanter for people and especially machines, Linked Research is going to implement a authoring tool to create and edit research papers directly within a browser. In addition it will enable the user to author triple statements to enrich the document with markup.

The objective of this thesis is to establish requirements for a user-friendly tool to create scholarly publications and the triple editing feature. The feature facilitates the enrichment of publications with metadata and enables scientists to make research knowledge more accessible to societies and more consumable for machines.

HTML

The Hypertext Markup Language (HTML) is the core language of the World Wide Web (WWW) and currently released as version 5.0. The language is used to create web pages and is recommended by the main international standards organization for the WWW (W3C, World Wide Web Consortium). HTML specifies the structure of a website with tags (like <html> ). The following listing shows the basic structure of an HTML 5 document:

The DOM (Document Object Model)

The DOM is a language neutral interface, which allows scripts to dynamically access and change the structure of an (HTML)-document. With the DOM it is possible for scripting languages like JavaScript to access HTML elements and rearrange, delete or add new elements to the HTML-document. The DOM makes DHTML (Dynamic HTML) possible. (http://www.w3.org/DOM/)

Web Browsers

Web browsers (or just browsers) are used to represent web pages on the World Wide Web. They provide the user interface for web applications. The current major browsers on the market are Firefox, Internet Explorer, Google Chrome, Opera and Safari with their corresponding web browser engines Blink (Google Chrome and Opera) Trident (Internet Explorer), Gecko (Firefox) and Webkit (Safari). The web browser engine is a software component that is responsible to display content on the screen.

Linked Data

Linked Data is part of the Semantic Web and describes free available structured data reachable through Uniform Resource Identifiers (URI). It is published in a format (often RDF) so that it can be interlinked with other data, and through that, be more useful. (Ca-padisli 2013).

Time Bernes-Lee published on 2006-07-27 an article called „Linked Data – Design Issues“. In this publication Bernes-Lee addresses four rules for Linked Data:

  1. Use URI as names for things.
  2. Use HTTP URIs so that people can look up those names.
  3. When someone looks up a URI, provide useful information, using the standards (RDF, SPARQL).
  4. Include links to other URIs. So that they can discover more things.

HTTP URI

As seen in Chapter 2.1, URIs help to refer to things and concepts on the web and build the basis of Linked Data. URIs are used to identify a name of a resource on the web. The HTTP scheme is the most common used form of URIs. They provide us a simple way to explore the web and its resources (Berners-Lee 2006).

RDF

RDF is a family of World Wide Web Consortium (W3C) specifications. In essence it is an entity-relationship model that lets anyone make any statement about anything. A statement is composed of three atomic parts – the subject, predicate and object. These parts together are called a triple. The subject (line 1 of listing 2) says what the statement is about and the predicate (line 2 of listing 2) describes the relationship to its object (line 3 of listing 2) resource. Typically triples are represented by HTTP URIs. The following triple statement gives an example:

1
2
3
<http://renatostauffer.ch/i#>
<http://xmlns.com/foaf/0.1/knows>
<http://csarven.cs/#i>

This statement says that the thing <http://renatostauffer.ch/i#> knows the thing <http://csarven.cs/#i> . Those things (resources) can now be further described with different RDF vocabularies. (RDF Working Group 2014). As the example from above shows, predicates are vocabulary terms that are used to relate the subject with an object. The predicate itself is an HTTP URI, which has a description for the term in use when dereferenced. (Capadisli 2012).

RDFa

Resource Description Framework in Attributes or simply RDFa is a W3C recommendation that enables authors of HTML, XHTML and other XML-based document types to enrich their documents with metadata. RDFa provides a set of attributes to describe elements in XML languages:

about
What is the resource about (subject in a triple statement)
rel / rev
Relationships between resources (predicate in a triple statement)
resource
Specifies the context (To which URL does the markup apply?)
src / href / resource

Requirement Engineering

In this chapter, the requirements for the authoring tool will be established. For this propose parts of the requirements engineering framework according to Pohl et all is used as a reference. The used framework is structured as shown in figure 1 (Fig. 1).

The framework is composed of three parts: The System Context, Core Activities and Requirements Artifact.

Firstly the System Context will be described to identify important characteristics of the system and give the reader of this thesis an overview of the system’s context. Depending on the established context, goals were determined and scenarios to be implemented are worked out. Those Requirement Artifacts are documented in the Chapters 5.3 Goals as functional and non-functional requirements and 5.4 Scenarios.

System Context

Each software-intensive system lies within a certain context and is through that influenced by the definition of the system requirements (Pohl et all, 2012). This chapter describes the system context with its facets for Linked Research. The Usage Facet, Subject Facet, IT-System Facet and Development Facet each describe a certain aspect of Linked Research and build the entry point for the systems requirements.

The system context was established based on meetings with Mr. Sarven Capadisli and studying the following artifacts:

  • A publication about Enabling Accessible Knowledge (http://csarven.ca/enabling-accessible-knowledge)
  • Linked Research Github repository (https://github.com/csarven/linked-research
  • Linked Research documentation (http://linked-research.270a.info/linked-research.html)
  • Linked Research publication (http://csarven.ca/linked-research)

Usage Facet

The Usage Facet describes how the user interacts with the system (Pohl et all, 2012). In the case of Linked Research the Usage Facet is described with the UML 2.0:

Figure 2 (Fig. 2) describes the ways an author or a consumer of a publication interacts with Linked Research. From a authors perspective the following use cases arise:

Edit Text
This is the most basic use case for Linked Research. The author creates or edits text directly in the browser.
Create Triple Statement
The author can enrich the written text with RDF triples via a triple authoring tool. For this to be working, the use cases Assign Subject, Assign Predicate and Assign Object need to be processed.
Remove Triple Statement
The author can remove added triple statements from the document.
Automatic Markup
A author does usually not have the time to enrich the whole publication with metadata by hand. Because of that, the use case Automatic Markup should provide the possibility to automatically enrich sections of the publication.
Local Storage
The text the author is editing gets stored locally. This way, off-line editing is possible.
Export Document
The author or consumer is able to export the document in HTML format or print the document (PDF format).
Change View
The author or consumer can apply different views for the document (LNCS ACM W3C-REC Native) Create an Abbreviation index at the und!
Assign Subject
In order to be able to create a triple statement, the author must have the ability to mark a text passage (a word, paragraph or the document itself) as an RDF subject.
Assign Predicate
A service should provide the author the ability to search for predicates in existing Linked Open vocabularies and assign a found predicate to a triple statement.
Assign Object
Finally the author needs functionality to assign a RDF object to a RDF triple statement

The use cases Change View, Local Storage and Export Document were already implemented into Linked Research by Sarven Carpadisli. The other use cases (Create Triple Statement, Remove Triple Statement and Automatic Markup, Edit Text, Assign Subject, Assign Predicate and Assign Object) are open to be implemented. The focus of the following chapters lies on the not yet implemented use cases from an author’s perspective.

Based on the established Usage Facet a first sketch of Linked Research was drawn:

  1. The user edits or creates text with the editor. When the user selects a text passage or word a popup menu shows up. From there the user has the ability to assign styles to the text (bold text, italic, etc.).
  2. The user can also toggle in / out the meta editor to create triple statements.
  3. The triple store is held locally. The user can insert or delete triple statements.
  4. Triple statements can be fetched to e.g. visualize the current graph.
  5. The meta editor talks to a external service to retrieve suggestions for possible RDF vocabulary terms when creating a triple statement.

Subject Facet

This facet describes what kind of objects must be represented by the system (Pohl et all, 2012). Table 3 describes objects and events within the system context that are important for the system.

Important objects and events within Linked Research
Object Description
RDF Subject An RDF Subject represents some part of the document the author writes. This can either be a single or multiple words, a single or multiple paragraphs or the hole document itself.
RDF Predicate A RDF predicate is the second part of a triple statement and defines the property for the RDF Subject. A predicate must always be a Uniform Resource Identifier.
RDF Object A Resource Description Framework (RDF) object is the third part of an RDF statement. Objects are frequently referred to as a property value that is mapped to a subject by the predicate. An object can be an RDF URI reference, an RDF literal, or a blank node. If the object is a blank node, it serves as a parent property to encapsulate a group of sub-properties.
RDF Triple Statement A triple statement is composed of a RDF Subject, RDF Predicate and RDF Object. Triple statements are stored within a RDF triple store (see next row)
RDF Triple Store An RDF triple store, persists RDF statements. The triple store is responsible for querying, inserting or deleting a triple statement.
MetaEditor object An object responsible to create new triple statements has to be provided.
API Handler An object to handle API calls to a service that provides access to different RDF vocabularies must be available. This enables the author to search for existing vocabularies.

IT System Facet

The IT system facet contains all aspects concerning the operational and technical environment oft the system. This also includes restrictions or guidelines (Pohl, 2010). The online documentation of Linked Research defines some design principles, which need to be taken into account when defining the requirements of Linked Research. This includes: [source]

  • Documents are human and machine friendly
  • Using the plain old semantic HTML marking process, with further semantic annotations using microformats and RDF.
  • No server required. Works on local machine.
  • No installation. No account creation.
  • A Web browser is the only requirement, whether it is a Line Mode Browser, or Firefox Nightly.
  • Licenses and rights: Apache License and CC0.

Development Facet

The development facet contains all aspects regarding the development process of the system, including guidelines and constraints, development tools or quality assurance methods.

For Linked Research a component based software engineering approach according to Sommerville (2007) is applied. This approach tries to reuse existing software and frameworks as much as possible and thus reduces development time for a much faster delivery. Figure 3 (Fig. 3) shows the different stages of this reuse-oriented approach (Sommerville, 2007).

Especially the editor and the triple store are parts of Linked Research that do not have to be developed from ground. There are a lot of existing projects, which provide such functionality. Chapter XXX and XXX show how the two components (editor and RDF triple store) got evaluated for Linked Research.

Github (a versioning tool) is used to manage the code base of Linked Research and prototypes.

The publication Enabling Accessible Knowledge defines in the chapter Conceptual Design design principles, to which Linked Research sticks to. Those are:

The Rule of Least Power
This means that when developing for the Web, the least powerful language should be chosen [source]. Capadisli et all state in in Enabling Accessible Knowledge the following: By design, HTML fulfils the principle of least power. As far as computer languages and their environments go, HTML is the least common denominator; it can be used virtually from any device; from command-line terminals, desktop browsers, smartwatches to cars, whether on or offline. Together with HTTP, the undeniable penetration of HTML, both on a technological and social level makes it a clear winner for information distribution.
